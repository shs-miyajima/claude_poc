# 設計レビューチェックリスト（AI 独立レビュー）— アンケート作成（設問作成・編集・削除・下書き/公開・代理作成）

> 目的: 設計の見落としは下流（テスト設計・実装）でのやり直しコストが最も大きい。
> 作成コンテキストを引き継がない**専用レビューサブエージェント**（`sdd-design-reviewer`、readonly）が、
> このチェックリストに沿って別視点で検証する。影響範囲（§5 `IMPACT-xx`）は記載の突合だけでなく、
> **実際のコードベースを検索して裏取り**する。
>
> 運用: `.claude/rules/sdd-workflow.md`「フェーズ 2 > 独立レビュー（承認確認前に必須）」を参照。
> このレビューは人間の承認を代替しない（自己承認の禁止の原則は変わらない）。

## 使い方

1. `02-design.md` の作成後、**承認確認を人間に提示する前**に、専用レビューサブエージェント
   `sdd-design-reviewer`（定義: `.claude/agents/sdd-design-reviewer.md`、readonly）へ
   `01-requirements.md` / `02-design.md` / 本チェックリストのパスを渡し、
   下記項目を ✓ / ✕ と理由で判定させる
2. ✕ の指摘のうち機械的に自動修正可能なもの（記載漏れ・ID 付与漏れ・「影響なし」の理由未記入等）は
   メインの AI がその場で修正する
3. 設計方針の変更・要件の解釈など判断が必要な指摘は自動修正せず、承認確認の提示に
   含めて人間に判断を仰ぐ
4. 判定結果と対応状況を「4. 総合判定」の実施記録に記入する

## 1. 要件カバレッジ

- [x] `01-requirements.md` の全ユースケース（§4 UC-xx）が、設計のどのクラス・画面で実現されるか特定できるか
- [x] バリデーション（§7）の全条件（複合 VAL はサブルール単位）が §6 検証ルール対応に記載され、
      検証箇所（FormRequest / Service 等のクラス・メソッド）が特定できるか
- [x] 全画面項目（§5.2）が Blade / JS の設計に含まれているか
- [x] 権限（§8）の制御箇所（ルート・ミドルウェア・画面出し分け）が設計に明記されているか
      （指摘1: 一覧のスコープ絞り込み記載漏れを修正済み）
- [x] 非機能要件（§9）の実現方法が設計に記載されているか（設計で対応しないものは理由があるか）
- [x] Laravel / フロント（Blade / Vite / JS）の責務分界が `01-requirements.md` の記載と一致しているか

## 2. 影響範囲（§5 IMPACT-xx）— コードベースとの突合

- [x] 変更対象のテーブル・モデルを参照する既存の画面・API・Service が `IMPACT-xx` として列挙されているか
      （**Grep で実際に検索して**、設計に載っていない参照箇所がないか確認する）
      （指摘2: `company/home.blade.php`（要件 §5.1 の遷移元）を IMPACT-11 として追加済み）
- [x] 共通部品（レイアウト・共通 JS・共有 Service）への波及が検討されているか
- [x] 「影響なし」とした項目に、検討した上での判断であることが分かる理由が書かれているか
      （未検討の放置がないか）
- [x] 既存 E2E（`tests/e2e_tests/tests/`）のセレクタ・データ前提への影響が検討されているか
- [x] Migration が既存データ・既存テーブルに与える影響（NOT NULL 追加・型変更等）が検討されているか

## 3. 設計品質

- [x] Controller は薄く、ビジネスロジックが Service に置かれているか
- [x] クラス名・メソッド名が命名規約（PSR-4 / `StudlyCase` / `camelCase`）に沿っているか
- [x] メソッド概要がシグネチャレベルで書かれ、人間がレビューできる粒度か
      （実装コードが書き込まれていないか）
- [x] Migration の内容（カラム・型・制約・インデックス）が要件と矛盾していないか
- [x] 画面・ルート・フロント（Blade / JS）の構成が過不足なく記載されているか

## 4. 総合判定

| 状態 | 条件 | 対応 |
|------|------|------|
| 指摘なし | 1〜3 のすべての項目が ✓ | そのまま承認確認を人間に提示する |
| 自動修正済み | ✕ はあったが機械的に修正できた | 修正内容を記載の上、承認確認を人間に提示する |
| 要判断 | 設計方針の変更等、自動修正できない ✕ が残る | 承認確認に残課題として明記し、人間に判断を仰ぐ（自己判断で解消したことにしない） |
