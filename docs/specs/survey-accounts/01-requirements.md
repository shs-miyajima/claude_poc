# 要件定義 — アンケートシステム アカウント基盤（認証・企業/部署/管理者/ユーザー管理）

> status: 01-requirements.status を参照（`draft` | `approved` | `rejected`）
>
> 本機能の `open-questions.md` は全 16 件 resolved（open 0 件）。
> システム全体の決定事項は `../survey-system/README.md` §4 および同 `open-questions.md` を参照。

## 1. 概要

アンケートシステム（親: `survey-system`）の基盤となる、認証と組織・アカウント管理を構築する。
共通ログイン（企業コード + メールアドレス + パスワード）、企業・部署・管理者・ユーザーの
登録/編集/削除（論理削除）、ユーザーの CSV 一括登録、スーパーユーザーの全体/個別企業の
画面切替を対象とする。後続の ②アンケート作成 ③回答 ④ダッシュボードは本基盤の上に構築する。

## 2. 用語集

| 用語 | 定義 |
|------|------|
| スーパーユーザー | 弊社社員。全企業を対象に企業・部署・管理者・ユーザーを管理し、画面切替で個別企業画面に入れる |
| 管理者 | 企業の人事部など一部の担当者。自社のユーザー・部署のみ管理できる |
| ユーザー | 企業の社員。本機能では管理対象（登録・編集・削除される側）。ログインしてアンケート回答（③）を行う |
| 企業 | 管理者・ユーザーが所属するテナント単位。データは企業単位に分離される |
| 企業コード | ログイン時に企業を特定するためのシステム採番コード。形式は「C + 4 桁連番」（C0001 から採番）。スーパーユーザーは空欄でログイン |
| 部署マスタ | 企業ごとに管理する部署の一覧。ユーザーの所属部署はここから選択する |
| 個別企業画面 | スーパーユーザーが特定企業を選択して表示する、当該企業の管理者が見る画面と同等の画面 |
| 削除（無効化） | 本システムの削除はすべて論理削除。無効化されたアカウントはログイン不可。データは物理削除しない（親 Q32）。再有効化で取り消せる（Q-15） |
| 再有効化 | 無効化済みエンティティを有効に戻す操作。無効化済みは編集不可で、再有効化のみ可能（Q-15） |

## 3. スコープ

### 対象

- [x] Laravel 管理画面
- [x] フロント（Blade / Vite / JavaScript / Tailwind CSS）
- [x] DB スキーマ変更（companies / departments 新設、users 拡張）

### 対象外

- アンケートの作成・回答・ダッシュボード（後続スペック ②③④ の範囲）
- メール送信（招待・通知）・本人によるパスワードリセット（親スペックでスコープ外確定）
- 本人によるプロフィール・パスワード変更（Q-09 で今回不要と確定。変更は管理者/スーパーユーザー経由）
- 既存の welcome 画面（`resources/views/welcome.blade.php`）の変更は行わない
  （ログイン画面への導線が必要な場合はフェーズ 2 で影響範囲 `IMPACT-xx` として明示する）
- composer の新規パッケージ追加（認証は Laravel 標準セッション認証を自前実装。親 Q28）

## 5. 画面・API 仕様

### 5.1 画面

URL・遷移の詳細はフェーズ 2（設計）で確定する。一覧画面はすべて 20 件/ページのページングとし、
有効・無効を問わず全件を表示して状態（有効 / 無効）を表示する（Q-15。ページングの母集団は
無効化済みを含む）。無効化済みの行には編集導線を出さず、「有効化」（再有効化）操作のみ表示する。

| 画面 | 利用ロール | 概要 |
|------|-----------|------|
| ログイン | 全ロール | 企業コード（スーパーユーザーは空欄）+ メールアドレス + パスワード |
| 全体（全企業）画面 | スーパーユーザー | 企業一覧（登録・編集・削除）+ 各企業の個別企業画面への導線（親 Q26） |
| 個別企業画面（切替後トップ） | スーパーユーザー | 当該企業の管理者向け画面と同等 + 管理者管理メニュー（スーパーユーザーにのみ表示） |
| 管理者一覧・登録・編集 | スーパーユーザー | 個別企業画面内に配置 |
| ユーザー一覧・登録・編集 | スーパーユーザー・管理者 | |
| ユーザー CSV 登録 | スーパーユーザー・管理者 | アップロード + 結果（成功件数 / エラー行一覧）表示 |
| 部署一覧・登録・編集 | スーパーユーザー・管理者 | |
| 管理者向けトップ | 管理者 | ログイン後の遷移先。メニュー構成はフェーズ 2 で確定 |

※ ユーザー（社員）ロールのログイン後画面（アンケート一覧）は ③survey-answer の範囲。
本機能ではログイン成功の遷移先としてのプレースホルダ画面までを対象とする。

### 5.3 API / ルート

フェーズ 2（設計）で確定する。方針:

- 全画面ともセッション認証必須（ログイン画面を除く）。未ログインはログイン画面へリダイレクト
- ルートはロール別にミドルウェアでアクセス制御する（§8 の権限マトリクスを正とする）
- スーパーユーザーの「個別企業画面」の対象企業はセッション（またはルートパラメータ）で
  保持する（方式はフェーズ 2 で確定）

### 5.4 フロント連携

- 本機能に SPA 的な非同期処理は想定しない（フォーム POST 中心。方式はフェーズ 2 で確定）
- CSV 登録は multipart フォームでアップロードし、結果（エラー行一覧）をサーバーサイドで描画する
- グラフ描画（Chart.js）は ④survey-dashboard の範囲であり本機能では使用しない

## 6. データ要件

列の詳細（型・制約）はフェーズ 2 で確定する。

| テーブル | 操作 | 概要 |
|----------|------|------|
| companies | CRUD（削除は論理） | 新設。企業名・企業コード（C + 4 桁連番で採番）・無効化フラグ相当 |
| departments | CRUD（削除は論理） | 新設。企業ごとの部署マスタ（部署名のみ・企業内一意） |
| users | CRUD（削除は論理） | 既存テーブルを拡張。ロール（スーパーユーザー/管理者/ユーザー）・所属企業（スーパーユーザーは無所属）・生年月日・入社年月日・性別・部署・無効化フラグ相当を追加 |
| sessions | R/W | 既存（Laravel 標準セッション）。変更しない |

- 初期データ: スーパーユーザー 1 名をシーダーで投入する（親 Q4）
- 無効化されたアカウント・企業のデータは物理削除しない（②〜④の回答データ参照を保証。親 Q32）
- メールアドレスの一意制約は「企業単位・無効化済みを含む」（Q-07）。スーパーユーザー
  （無所属）同士も互いに一意とする

## 7. バリデーション・エラー

エラーコードは HTTP ステータス。`{項目名}` `{n}` は該当項目・値で置き換える。

| ID | 条件 | エラーコード | メッセージ |
|----|------|-------------|-----------|
| VAL-01 | ログイン: メールアドレスまたはパスワードが未入力 | 422 | {項目名}は必須です。 |
| VAL-02 | ログイン: 企業コード・メールアドレス・パスワードの組で認証失敗（存在しない・企業コードの形式不正/最大長超過・不一致・無効化済みアカウント・無効化済み企業） | 422 | 企業コード、メールアドレス、またはパスワードが正しくありません。 |
| VAL-03 | 企業名: 未入力 / 100 文字超過 | 422 | 企業名は必須です。/ 企業名は100文字以内で入力してください。 |
| VAL-04 | 氏名: 未入力 / 100 文字超過 | 422 | 氏名は必須です。/ 氏名は100文字以内で入力してください。 |
| VAL-05 | メールアドレス: 未入力 / 形式不正 / 255 文字超過 | 422 | メールアドレスは必須です。/ メールアドレスの形式が正しくありません。/ メールアドレスは255文字以内で入力してください。 |
| VAL-06 | メールアドレス: 同一企業内で重複（無効化済みアカウントとの重複を含む・Q-07） | 422 | このメールアドレスは既に登録されています。 |
| VAL-07 | パスワード: 未入力（登録時のみ。編集時は空欄可・Q-08）/ 8 文字未満 / 255 文字超過 | 422 | 初期パスワードは必須です。/ パスワードは8文字以上で入力してください。/ パスワードは255文字以内で入力してください。 |
| VAL-08 | 生年月日・入社年月日: 未入力 / 日付として不正 | 422 | {項目名}は必須です。/ {項目名}はYYYY-MM-DD形式の正しい日付で入力してください。 |
| VAL-09 | 性別: 未選択 / 選択肢（男性・女性・その他）以外の値 | 422 | 性別は必須です。/ 性別が正しくありません。 |
| VAL-10 | 部署: 未選択 / 対象企業の有効な部署マスタに存在しない。ただし編集時に変更前の所属部署のまま保存する場合に限り、無効化済み部署でも許容する（Q-16） | 422 | 部署は必須です。/ 部署が正しくありません。 |
| VAL-11 | 部署名: 未入力 / 100 文字超過 / 企業内で重複（無効化済みの部署との重複を含む・Q-14） | 422 | 部署名は必須です。/ 部署名は100文字以内で入力してください。/ この部署名は既に登録されています。 |
| VAL-12 | CSV: ファイル未選択 / CSV 形式でない / UTF-8 でない | 422 | CSVファイルを選択してください。/ CSVファイル（UTF-8）の形式が正しくありません。 |
| VAL-13 | CSV: データ行が 0 件 / 1,000 件超過 | 422 | CSVのデータ行は1件以上1,000件以下にしてください。 |
| VAL-14 | CSV: ヘッダー行が定義（Q-03 の 7 列）と不一致 / 列数不足 | 422 | CSVのヘッダー行が正しくありません。 |
| VAL-15 | CSV: 行単位の検証エラー（VAL-04〜VAL-10 と同一基準）。1 行でもエラーがあれば全件ロールバックし、行番号 + 理由を一覧表示 | 422 | {行番号}行目: {VAL-04〜VAL-10 の該当メッセージ} |
| VAL-16 | CSV: ファイル内でメールアドレスが重複 | 422 | {行番号}行目: メールアドレスがファイル内で重複しています。 |
| VAL-17 | CSV: データ行の列数が定義（Q-03 の 7 列）と不一致（エラー行扱い・全件ロールバック対象） | 422 | {行番号}行目: 列数が正しくありません。 |

## 8. 権限

○=可 / ×=不可（URL 直接アクセスは 403。メニュー表示の有無はフェーズ 2 で確定）。
スーパーユーザーの企業・部署・管理者・ユーザー操作は全企業に対して可、管理者は自社のみ。

| 操作 | スーパーユーザー | 管理者 | ユーザー |
|------|-----------------|--------|----------|
| ログイン / ログアウト | ○ | ○ | ○ |
| 企業の登録・編集・削除・再有効化 | ○ | × | × |
| 全体 / 個別企業の画面切替（有効な企業のみ・Q-15） | ○ | × | × |
| 管理者の登録・編集・削除・再有効化 | ○（個別企業画面） | × | × |
| ユーザーの登録・編集・削除・再有効化 | ○（個別企業画面） | ○（自社のみ） | × |
| ユーザー CSV 登録 | ○（個別企業画面・親 Q29） | ○（自社のみ） | × |
| 部署マスタの登録・編集・削除・再有効化 | ○（個別企業画面） | ○（自社のみ） | × |
| 他企業のデータへのアクセス | ○（全企業） | × | × |

## 9. 非機能要件

性能目標は親 Q27 でスコープ外（dropped）とされたため定義しない。

| ID | 非機能要件 | 備考 |
|----|-----------|------|
| NFR-01 | パスワードは平文保存せずハッシュ化して保存する | Laravel 標準（bcrypt）。CSV の初期パスワードも同様 |
| NFR-02 | 未ログイン状態で認証必須 URL にアクセスした場合、ログイン画面へリダイレクトする | 全画面共通 |
| NFR-03 | ロール外・他企業の URL への直接アクセスは 403 で拒否する（画面にリンクを出さないだけの制御にしない） | §8 の権限マトリクスを正とする |
| NFR-04 | フォーム送信は CSRF 対策を行う | Laravel 標準 |
| NFR-05 | 認証失敗時のメッセージからアカウントの存在有無を判別できないようにする | VAL-02 と対応 |

## 10. 具体例（入出力サンプル）

### 正常系の例

1. スーパーユーザーが企業名「株式会社サンプル」で企業登録
   → 企業一覧に「株式会社サンプル」と採番された企業コード「C0001」が表示される
2. C0001 の個別企業画面で部署「人事部」を登録後、次の CSV（データ 1 行）を登録
   ```csv
   氏名,メールアドレス,初期パスワード,生年月日,入社年月日,性別,部署名
   山田太郎,taro.yamada@example.co.jp,pass1234,1990-04-01,2015-04-01,男性,人事部
   ```
   → 「1件登録しました」相当の完了表示となり、ユーザー一覧に「山田太郎」が表示される
3. 山田太郎が企業コード「C0001」+ メールアドレス「taro.yamada@example.co.jp」+
   パスワード「pass1234」でログイン
   → ログインに成功し、ユーザー向けトップに遷移する

### 異常系の例

1. ユーザー登録で初期パスワードに「pass123」（7 文字）を入力して送信
   → 「パスワードは8文字以上で入力してください。」（VAL-07）が表示され、登録されない
2. データ 5 行中 4 行目のメールアドレスが同一企業の既存ユーザーと重複する CSV を登録
   → 1 件も登録されず（全件ロールバック）、「4行目: このメールアドレスは既に登録されています。」
   （VAL-15・VAL-06 相当）が一覧表示される
3. 無効化済みの山田太郎が正しい企業コード・メールアドレス・パスワードでログイン
   → 「企業コード、メールアドレス、またはパスワードが正しくありません。」（VAL-02）が表示され、
   ログインできない

## 11. 受け入れ条件

| ID | 受け入れ条件 |
|----|-------------|
| AC-01 | 管理者・ユーザーが正しい企業コード + メールアドレス + パスワードでログインすると、それぞれのロール別トップ画面に遷移する |
| AC-02 | スーパーユーザーが企業コード空欄 + 正しいメールアドレス + パスワードでログインすると、全体（全企業）画面に遷移する |
| AC-03 | 誤ったパスワード・存在しない組み合わせ・無効化済みアカウントでのログインは、VAL-02 の統一メッセージが表示されログインできない |
| AC-04 | ログアウトすると、直前まで見ていた認証必須画面に戻ってもログイン画面へリダイレクトされる |
| AC-05 | 未ログインで認証必須 URL に直接アクセスすると、ログイン画面へリダイレクトされる |
| AC-06 | ユーザーのアカウントで管理者・スーパーユーザー用 URL に直接アクセスすると 403 になる |
| AC-07 | 管理者が他企業のユーザー管理 URL に直接アクセスすると 403 になる |
| AC-08 | スーパーユーザーが企業を登録すると、一覧に企業名と採番済み企業コード（C + 4 桁連番）が表示される |
| AC-09 | スーパーユーザーが企業を削除（無効化）すると、配下の管理者・ユーザーはログインできなくなる |
| AC-10 | スーパーユーザーが全体画面から企業を選択すると個別企業画面に切り替わり、当該企業のデータのみが表示され、全体画面に戻れる |
| AC-11 | スーパーユーザーが登録した管理者は、当該企業の企業コードでログインし管理者向けトップに遷移できる |
| AC-12 | 管理者（またはスーパーユーザーの個別企業画面）がユーザーを登録すると、ユーザー一覧に表示され、当該ユーザーはログインできる |
| AC-13 | ユーザーを編集（パスワード欄空欄）すると、変更後の値が一覧・編集画面に反映され、パスワードは変わらない（変更前のパスワードでログインできる） |
| AC-14 | ユーザーを削除（無効化）すると、当該ユーザーはログインできなくなる |
| AC-15 | 全行正常な CSV（N 件・N ≦ 1,000）を登録すると、N 件全員がユーザー一覧に表示される |
| AC-16 | エラー行を含む CSV を登録すると、1 件も登録されず、エラー行の行番号と理由が一覧表示される |
| AC-17 | データ行が 1,001 件の CSV を登録すると、VAL-13 のエラーが表示され 1 件も登録されない |
| AC-18 | 部署を登録すると、ユーザー登録・編集画面の部署選択肢に表示される |
| AC-19 | シーダー投入されたスーパーユーザーでログインできる |
| AC-20 | ユーザーが 21 名（有効・無効を問わず）登録されている場合、一覧の 1 ページ目に 20 件が表示され、2 ページ目に残り 1 件が表示される |
| AC-21 | 所属ユーザーがいる部署を削除（無効化）でき、以後のユーザー登録・編集画面の部署選択肢に表示されず、既存所属ユーザーの編集画面・一覧の所属表示は維持される |
| AC-22 | 無効化済みのユーザーを再有効化すると、当該ユーザーは再びログインできる |
| AC-23 | 無効化済みの企業を再有効化しても、配下の管理者・ユーザーは無効のまま（ログイン不可）である |
| AC-24 | 無効化済みの企業は全体画面の一覧に「無効」の状態表示付きで表示され、個別企業画面への切替導線が表示されない |
| AC-25 | 無効化済みの部署に所属するユーザーを、部署を変更せずに編集（例: 氏名の変更）して保存でき、変更内容が反映される |
| AC-26 | 無効化済みの企業・管理者・ユーザー・部署の編集 URL に直接アクセスすると 403 になる（一覧の編集導線も表示されない） |
