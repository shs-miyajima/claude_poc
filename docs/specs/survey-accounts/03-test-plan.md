### 1.1 レイヤ分担

| レイヤ | ツール | 役割 | 本機能での対象 |
|--------|--------|------|----------------|
| 上（少） | Playwright E2E | 重要ジャーニー **正常系** ＋ **ビジネスクリティカル異常系** | ログイン（super/admin/user）・企業/管理者/ユーザー/部署の代表登録ジャーニー・企業切替・CSV一括登録・無効化のconfirm()挙動・ページング操作（計 17 件） |
| 中（多） | PHPUnit | **単体**（ロジック・境界値・異常系）＋ **結合**（DB・Service 連携） | AuthenticationService／CompanyService／AdminService／UserService／DepartmentService／UserCsvImportService／FormRequest 全種（Update 系含む）／HTTP 認可（ロール×URL）／Model・Enum／NFR／回帰確認（計 136 件） |
| 中（多） | Vitest | JS **単体**（純関数・状態管理） | 該当なし（02-design.md §3.2 のとおり本機能では新規 JS モジュールを追加しないため） |

| 種別 | 対象 | ツール | 備考 |
|------|------|--------|------|
| E2E | UC 主要フロー、権限、ジャーニー上の UI | Playwright | `inp` は各画面 1 スモーク（企業/管理者/ユーザー/部署の登録画面）＋クリティカル異常（ログイン失敗・CSV エラー表示） |
| 単体（サーバー） | Service / FormRequest / Enum / Model | PHPUnit | `layer:unit`（FormRequest 含む） |
| HTTP 検証 | POST 等 → 422 / エラー | PHPUnit | `layer:http`（Controller 単体は作らない） |
| 結合（サーバー） | Service + DB | PHPUnit | CSV 備考に `layer:integration` |
| 単体（フロント） | JS 純関数 | Vitest | 対象 JS モジュールなし |

### 1.2 E2E に載せないもの（PHPUnit / Vitest へ）

- 境界値の NG/OK ペア（文字数・行数・件数等）はすべて PHPUnit に配置（§1.3・§1.4）
- 複合 VAL（VAL-02・VAL-15 等）のエラー理由ごとの網羅は PHPUnit に配置
- ロール×URL の 403 網羅（NFR-03・AC-06・AC-07・AC-26）は PHPUnit HTTP に配置し、E2E は画面挙動（ログイン失敗・未ログインリダイレクト）のみ
- 絞り込み・集計（部署選択肢反映等）の条件分岐は PHPUnit（結合）に配置し、E2E は代表 1 ジャーニーのみ

### 期待結果の書き方（曖昧表現の排除）

全 CSV の期待結果は、件数・文言・画面状態など検証可能な具体値で記載済み（「正しく」「適切に」「問題なく」等は使用していない）。

### 独立性・再現性の標準観点

- 各 PHPUnit ケースは `RefreshDatabase` を用いた Feature/Unit テストとして、ケースごとに前提データ（企業・ユーザー・部署等）をテスト内で新規作成する。他ケースが作成したレコードを暗黙の前提にしない
- 一意制約（メール・部署名・企業コード）が関わるケースは、ケースごとに固有の値（例: `dup@a`、`dup2@a` 等）を用いて他ケースと衝突しないようにする
- E2E は `migrate:fresh --seed` 直後の状態を前提とし、Seeder 由来の初期データ（スーパーユーザー）を参照するのみで更新しない。企業・ユーザー・部署の作成が必要なケースはテスト内で新規作成する
- AC-20／PU-112／E2E-017 のような 21 件データを前提とするケースは、テスト（または E2E 実行前セットアップ）内で当該ケース専用の企業・ユーザーを作成し、他ケースの一覧件数に影響しないようにする

#### 並列実行安全性（E2E は並列実行が前提）

- E2E の各ケースは、操作対象のレコード（企業・管理者・ユーザー・部署）をケースごとに新規作成してから更新・無効化する（Seeder の初期データを直接更新しない）
- E2E-012（ユーザー無効化）は当該テスト内で作成した専用ユーザーを対象とし、他ケースが参照する既存ユーザーには影響しない
- E2E-017（ページング）は当該テスト専用に 21 件のユーザーを作成し、他ケースが作成するユーザー数に依存しない前提（テスト対象企業を専用に用意する）とする

### CSV ケース粒度の標準観点

各 CSV は 1 行 1 観点。VAL の複合ルールはサブルールごとに、エンティティ（管理者/ユーザー/CSV）ごとに、ロール・権限パターンごとに個別行として作成済み（§1.3・§1.4 参照）。

※ パスワードは Q-08 により「空欄なら変更しない」仕様が要件定義（01-requirements.md §5.2）に明記されている。他の任意項目（空入力で上書き）は本機能に存在しない（部署のみ Q-16 により「変更前の無効化済み所属を維持する」特殊ケースがあり、PU-081 で確認済み）。

## 4. テストデータ

- **PHPUnit**: `CompanyFactory` / `DepartmentFactory` / `UserFactory`（`superUser()` / `admin()` / `deactivated()` / `forCompany(Company)` state を使用）でケースごとに前提データを生成する。CSV 取込テストは `tests/Fixtures/` 等にサンプル CSV を配置するか、テスト内で一時ファイルを生成する
- **E2E**: `migrate:fresh --seed` 実行直後の状態を前提とする。Seeder 投入内容は `SuperUserSeeder`（`.env` の `SUPER_USER_EMAIL` / `SUPER_USER_PASSWORD`）のみ。企業・管理者・ユーザー・部署は各 E2E ケース内で作成する
