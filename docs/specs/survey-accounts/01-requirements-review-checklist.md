# 要件レビューチェックリスト（AI 独立レビュー）— アンケートシステム アカウント基盤

> 目的: `01-requirements.md` §13 のセルフチェックは**作成者（AI）の自己確認**であり、
> 作成者自身の見落としには気づけない。作成コンテキストを引き継がない**専用レビュー
> サブエージェント**（`sdd-requirements-reviewer`、readonly）が、このチェックリストに沿って
> 別視点で検証する。
>
> **AI の主対象**: 仕様完全性・内部整合性・検証可能性。業務妥当性の最終判断は人間が行う。
>
> 運用: `.claude/rules/sdd-workflow.md`「フェーズ 1 > 独立レビュー（承認確認前に必須）」を参照。
> このレビューは人間の承認を代替しない（自己承認の禁止の原則は変わらない）。

## 使い方

1. `01-requirements.md` の確定案作成後、**承認確認を人間に提示する前**に、専用レビューサブエージェント
   `sdd-requirements-reviewer`（定義: `.claude/agents/sdd-requirements-reviewer.md`、readonly）へ
   `01-requirements.md` / `open-questions.md` / 本チェックリストのパスを渡し、
   下記項目を ✓ / ✕ と理由で判定させる
2. ✕ の指摘のうち機械的に自動修正可能なもの（ID 欠番、§5.2↔§7 不一致、曖昧語、
   open-questions 未解決のままの確定表記等）はメインの AI がその場で修正する
3. 仕様の解釈・業務ルールの変更を伴う指摘は自動修正せず、承認確認の提示に
   含めて人間に判断を仰ぐ
4. エージェント出力の**判定表（全項目）**は、監査用に実施記録の下（「判定表転記」節）へ貼り付ける。
   実施記録 1 行の「指摘事項と対応」は要約とし、詳細は判定表を正とする
5. `人間（業務）` 分類のコメントはチェックリストの ✕ 件数・総合判定に**含めない**。
   実施記録の「指摘事項と対応」に `【参考（業務）】` として転記し、
   必要なら人間への承認確認に参考質問として記載する（仕様の自動修正は行わない）
6. 判定結果と対応状況を「6. 総合判定」の実施記録に記入する

## 0. 完成度・未解決事項

- [ ] `open-questions.md` に **未解決（resolved でない）** の質問が残っていないか
- [ ] 本文に【仮定】等の仮置き表記が残っていないか（仮定での仮置きは運用禁止。残っていれば ✕ とし、
      `open-questions.md` への起票と回答の反映を求める）
- [ ] 本文に、根拠のない確定記述（`open-questions.md` の回答・§12 参照資料のいずれにも根拠がない仕様）が残っていないか
- [ ] §3 対象・対象外が両方記載され、対象外に「変更してはいけない既存機能」が明記されているか
- [ ] §13 承認依頼前セルフチェックの全項目が本文上も満たされているか（チェックボックスだけ ✓ で実体が欠けていないか）

## 1. 内部整合性（節横断）

- [ ] §5.2 の全行について、型・必須/任意・最大長が空欄のまま残っていないか
      （ボタン等の非入力行は「-」で明示。表示専用項目は型と「該当なし」等の根拠があること）
- [ ] §5.2 画面項目の「必須」「最大長」「型」と §7 VAL が矛盾なく対応しているか
      （必須項目に VAL がない、最大長が不一致、型と検証条件が食い違う行がないか）
- [ ] §4 UC の操作が §5.1 画面・§5.3 ルートで実現可能な導線として記載されているか（UC に対応する画面・URL の欠落がないか）
- [ ] §8 権限に、§4 の全操作 × 全ロールの可否が記載されているか（暗黙の許可・禁止がないか）
- [ ] §6 データ要件（削除方針・一意制約等）と §4 UC・§5 の挙動が矛盾していないか
      （論理削除なのに物理削除の記述、削除後の参照ルールの不一致等）
- [ ] §3 対象外と §4 UC・§5 画面が矛盾していないか（対象外と書いた機能が UC に含まれていないか）
- [ ] §9 NFR と §4 UC・§8 権限が矛盾していないか
      （例: マルチテナント分離 NFR があるのに他社データアクセスの拒否 UC・権限がない）
- [ ] §2 用語集の定義と本文の使い方が一致しているか（同一語で異なる意味を使っていないか）

## 2. 検証可能性

- [ ] §11 AC がすべて「具体的な操作 → 期待結果」の検証可能な文になっているか
      （「〜できること」だけ、期待結果が曖昧な文がないか）
- [ ] §7 VAL・§10 具体例・§11 AC に「正しく」「適切に」「問題なく」等の曖昧語がないか
- [ ] §10 に正常系・異常系が実際の値で 1 件以上ずつ書かれているか
- [ ] 数値境界（件数・文字数・設問数等）がある項目で、§10 または §7 に **両側**
      （ちょうど OK / 超過 NG）が書かれているか
- [ ] エラーケースでエラーメッセージが確定文言またはパターンとして書かれているか
      （「エラーを表示」のみで終わっていないか）

## 3. ID・トレーサビリティ準備

- [ ] UC / VAL / AC / NFR に一意な ID が付与され、重複がないか
      （UC の派生 ID 例: `UC-13a` は親 UC `UC-13` の分岐として許容。同一親の欠番は不可）
- [ ] §5.2 の全入力項目について、対応する VAL または「任意・検証なし」の根拠があるか
- [ ] 状態を持つエンティティ（下書き/公開/終了等）について、状態ごとの操作可否が UC または §5 に書かれているか

## 4. 非機能・環境制約

- [ ] §9 NFR が「特になし」の場合、性能・ログ・セキュリティ等を検討した上での判断であることが読み取れるか
      （未検討のまま空欄・特になしになっていないか）
- [ ] セキュリティ関連（認証・認可・テナント分離・入力の危険な型）が要件として明示されているか
      （該当機能がある場合）
- [ ] §9・§1 の環境制約が `AGENTS.md` / `docker-compose.yml` と矛盾していないか

## 5. 既存資産・スコープ（コードベース裏取り）

- [ ] §3 や §5 で変更対象とした既存画面・ルートが、コードベースに実在するか
- [ ] §3 対象外とした既存機能を変更する記述が §4〜§5 に紛れ込んでいないか
- [ ] §12 参照資料（元 `.spec` 等）がある場合、意図的な差分以外の記載漏れがないか

## 6. 総合判定

| 状態 | 条件 | 対応 |
|------|------|------|
| 指摘なし | **0〜5** のすべての項目が ✓（`人間（業務）` の参考コメントの有無は問わない） | そのまま承認確認を人間に提示する |
| 自動修正済み | ✕ はあったが機械的に修正できた | 修正内容を記載の上、承認確認を人間に提示する |
| 要判断 | 仕様解釈・業務方針の変更等、自動修正できない ✕ が残る | 承認確認に残課題として明記し、人間に判断を仰ぐ（自己判断で解消したことにしない） |

`人間（業務）` は上記の ✕ 件数・総合判定に含めない。実施記録の `【参考（業務）】` に記載する。

### 実施記録の書き方

「指摘事項と対応」列は次の 3 ブロックに分けて記載する（該当なしのブロックは省略可）。

- `【仕様指摘】` — チェックリストの ✕ と対応（`機械修正可` / `要判断`）
- `【参考（業務）】` — エージェントの `人間（業務）` 分類。仕様の ✕ には含めない
- `【対応】` — 修正内容・残課題・承認確認への転記の要約
