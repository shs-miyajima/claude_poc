# 設計レビューチェックリスト（AI 独立レビュー）— {{display_name}}

> 目的: 設計の見落としは下流（テスト設計・実装）でのやり直しコストが最も大きい。
> 作成コンテキストを引き継がない**独立したレビュー用サブエージェント**（`sdd-design-reviewer`）が、
> このチェックリストに沿って別視点で検証する。影響範囲（IMPACT）は記載の突合だけでなく、
> **実際のコードベースを検索して裏取り**する。
>
> 運用: `.claude/rules/sdd-workflow.md`「フェーズ 2 > 独立レビュー（承認確認前に必須）」を参照。
> このレビューは人間の承認を代替しない（自己承認の禁止の原則は変わらない）。

## 使い方

1. `02-design.md` の作成後、**承認確認を人間に提示する前**に、独立レビュー用の
   サブエージェント（`subagent_type: sdd-design-reviewer`、readonly）へ
   `01-requirements.md` / `02-design.md` / 本チェックリストのパスを渡し、
   下記項目を ✓ / ✕ と理由で判定させる
2. ✕ の指摘のうち機械的に自動修正可能なもの（記載漏れ・ID 付与漏れ・「影響なし」の理由未記入等）は
   メインの AI がその場で修正する
3. 設計方針の変更・要件の解釈など判断が必要な指摘は自動修正せず、承認確認の提示に
   含めて人間に判断を仰ぐ
4. 判定結果と対応状況を「5. 総合判定」の実施記録に記入する

## 1. 要件カバレッジ

- [ ] `01-requirements.md` の全 UC が、設計のどのクラス・画面で実現されるか特定できるか
- [ ] 全 VAL（§7）に対応する検証箇所（Form Request 等）が設計に含まれているか
- [ ] 全画面項目（§5.2）が Blade / JS の設計に含まれているか
- [ ] 権限（§8）の制御箇所（ルート・ミドルウェア・画面出し分け）が設計に明記されているか
- [ ] NFR（§9）の実現方法が設計に記載されているか（設計で対応しないものは理由があるか）
- [ ] Laravel / フロント（Blade / Vite / JS）の責務分界が `01-requirements.md` の記載と一致しているか

## 2. 影響範囲（IMPACT）— コードベースとの突合

- [ ] 変更対象のテーブル・モデルを参照する既存の画面・API・Service が IMPACT に列挙されているか
      （**Grep で実際に検索して**、設計に載っていない参照箇所がないか確認する）
- [ ] 共通部品（レイアウト・共通 JS・共有 Service）への波及が検討されているか
- [ ] 「影響なし」とした項目に、検討した上での判断であることが分かる理由が書かれているか
      （未検討の放置がないか）
- [ ] 既存 E2E（`tests/e2e_tests/tests/`）のセレクタ・データ前提への影響が検討されているか
      （既存のワイルドカードセレクタ `^=` / `*=` と新規 `data-testid` の衝突を含む）
- [ ] Migration が既存データ・既存テーブルに与える影響（NOT NULL 追加・型変更等）が検討されているか

## 3. 設計品質

- [ ] Controller は薄く、ビジネスロジックが Service に置かれているか
- [ ] クラス名・メソッド名が命名規約（PSR-4 / `StudlyCase` / `camelCase`）に沿っているか
- [ ] メソッド概要がシグネチャレベルで書かれ、人間がレビューできる粒度か
      （実装コードが書き込まれていないか）
- [ ] Migration の内容（カラム・型・制約・インデックス）が要件と矛盾していないか
- [ ] 全 FK カラムにインデックスが設計されているか（PostgreSQL は FK に自動で張らないため、
      §2.4 インデックス表に FK 行が漏れなくあるか）
- [ ] 業務上の一意制約（VAL-xx）が unique index として §2.4 に含まれているか
      （アプリ側バリデーションだけに頼っていないか）
- [ ] 要件の検索・絞り込み・並び順（UC / §5.2）に対応するインデックスが §2.4 の表にあり、
      根拠 ID が書かれているか（張らないと判断した検索カラムに理由が書かれているか）
- [ ] 根拠 ID のないインデックス・単独の低カーディナリティ列へのインデックスがないか
      （ある場合、設計判断の理由が書かれているか）
- [ ] 複合インデックスの列順が「等値条件が先、範囲・ORDER BY が後」になっているか
- [ ] 画面・ルート・フロント（Blade / JS）の構成が過不足なく記載されているか
- [ ] E2E が操作・検証する要素への `data-testid` 付与方針が記載されているか

## 4. 非機能設計（§5 との突合）

- [ ] §5.3 セキュリティ観点チェックの全項目が、チェックを付けるだけでなく**担保箇所
      （どの層・どのクラス・どの仕組みか）を特定できる記載**になっているか
      （機械的な ✓ で流していないか）
- [ ] 認可（ロール・所有者判定）の担保層が、要件の権限（§8）と矛盾していないか
- [ ] ファイルアップロードがある場合、検証（拡張子・MIME・サイズ）と保存先
      （公開ディレクトリ外）が設計されているか（「なし」の場合、その判断が妥当か）
- [ ] ログ・エラーレスポンスに機微情報（個人情報・スタックトレース）を出さない方針と
      担保箇所が書かれているか
- [ ] 一覧・参照系の Service メソッドに Eager Loading（`with()`）対象と
      ページネーション方針が書かれているか（N+1 の予防。§2.2）
- [ ] 削除方針（論理/物理）と FK の削除時挙動（cascade / restrict / set null）が
      §2.4 の表に記載されているか（論理削除の場合、unique 制約との干渉が検討されているか）
- [ ] トランザクション・競合（§5.1）とエラーハンドリング（§5.2）の表が空欄のまま
      放置されていないか（該当なしの場合、理由が書かれているか）

## 5. 総合判定

| 状態 | 条件 | 対応 |
|------|------|------|
| 指摘なし | 1〜4 のすべての項目が ✓ | そのまま承認確認を人間に提示する |
| 自動修正済み | ✕ はあったが機械的に修正できた | 修正内容を記載の上、承認確認を人間に提示する |
| 要判断 | 設計方針の変更等、自動修正できない ✕ が残る | 承認確認に残課題として明記し、人間に判断を仰ぐ（自己判断で解消したことにしない） |
