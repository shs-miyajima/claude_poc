# 仕様駆動開発ワークフロー

対象: 常時適用。返答は日本語。指定された作業範囲以外のコードは修正しない。

## 正本

- 実装・テストの判断は `docs/specs/` を正とする
- 仕様は Markdown で管理する（`docs/specs/<slug>/`）
- 外部資料（チャット入力等）と specs が食い違う場合は、実装に進まず差分を報告する

## 長時間セッションでの注意（コンテキスト要約への備え）

1 機能を 1 チャットで通す運用のため、会話が長くなるとコンテキストの要約（圧縮）が発生し、
読み込み済みの Skill 本文が失われることがある。

- **フェーズを切り替えるとき、および要約発生後に作業を再開するとき**は、その作業に該当する
  ドメイン規約 Skill（`laravel-conventions` / `frontend-vite-tailwind` / `testing-playwright` /
  `testing-vitest`）を再度読み込んでから作業する
- 進捗の正本はチャット履歴ではなく `docs/specs/<slug>/` のファイル（`*.status`・`changelog.md`）。
  要約後は記憶に頼らず、まず status ファイルを読み直して現在フェーズを確認する

## 機能 ID（TBD 運用）

- 命名規則は未確定。新規は `docs/specs/<slug>/` で開始（slug は英小文字・ハイフン推奨）
- `meta.yaml` の `display_name` に日本語の正式名称を書く
- E2E との対応は `docs/specs/_registry.md` に追記する

## フェーズと停止条件

各フェーズ完了後は**停止**し、人間の承認を待つ。承認なしに次フェーズへ進まない。

| フェーズ | 成果物 | 承認ファイル |
|---------|--------|-------------|
| 1. 仕様整理 | `01-requirements.md`, `open-questions.md`, `effort-report.md`（見積） | `01-requirements.status` |
| 2. 設計 | `02-design.md`, `02-design-review-checklist.md`（AI 独立レビュー結果） | `02-design.status` |
| 3. テスト設計 | `03-test-plan.md`, `03-test-plan.csv`, `03-test-plan-phpunit.csv`（該当時）, `03-test-plan-vitest.csv`（該当時）, `03-test-plan-review-checklist.md`（AI 独立レビュー結果） | `03-test-plan.status` |
| 4. 実装・テスト | コード・テストコード, `04-completion-report.md` | — |

`*.status` の値: `draft` | `approved` | `rejected`

### status ファイルの形式

1 行目に状態、2 行目以降に記録を書く。

```
approved
date: YYYY-MM-DD
phase: 01-requirements
```

### 承認の判定（自己承認の禁止）

- 各フェーズの成果物が完成したら、**選択式の承認確認（承認 / 差戻し）を必ず提示して停止する**。
  提示には `AskUserQuestion` ツールを使い、「承認」「差戻し（理由を入力）」の選択肢を出す
  （地の文で「よろしいですか？」と聞くだけの承認確認はしない）
- `*.status` を `approved` に変更できるのは、**直近のユーザー発言に明示的な承認がある場合のみ**（例: 承認ボタンの選択、「承認」「フェーズ1承認」「次へ進んで」）
- 過去の会話の解釈・エージェント自身の判断・成果物が完成したという事実を理由に `approved` へ変更してはならない
- 「OK」「いいね」など対象フェーズが曖昧な返答の場合は、**どのフェーズへの承認かを確認してから**更新する
- `approved` になるまで次フェーズのファイル作成・コード編集・テスト実行をしない

### 記録

- **承認時・差戻し時のいずれも** `changelog.md` に 1 エントリ記録する（差戻し時は理由も必須）
- 差戻し時は `rejected` にする
- `*.status`・`changelog.md`・`effort-report.md` に記入する日付・時刻は、**推測で書かず**
  Bash ツールの `date` コマンドで取得した現在時刻を使う（工数計測の信頼性に直結するため）

### Git 運用（フェーズコミット）

チャット履歴の外に**復元ポイントと監査証跡**を残すため、以下のタイミングで必ず git commit する。
特にフェーズ 3 承認時のコミットは、フェーズ 4（自走）が失敗したときに
「テスト設計承認時点に戻す」ための復元ポイントになる。

- **各フェーズの承認直後**: そのフェーズの成果物と `*.status`・`changelog.md`・`effort-report.md` の
  更新をまとめてコミットする
- **フェーズ 4 の完了報告作成後**: 実装・テストコード・`04-completion-report.md` をコミットする
- コミットメッセージ形式: `[SDD][<slug>] フェーズN(<フェーズ名>) <操作>`
  - 例: `[SDD][csv-import] フェーズ1(仕様整理) approved`
  - 例: `[SDD][csv-import] フェーズ4(実装・テスト) 完了`
- コミット前に `git status` と `git diff` で対象を確認し、**当該機能に関係のない変更を混ぜない**
- `git push` は自動では行わない（ユーザーの指示があった場合のみ）

### 承認後の仕様変更（差分承認）

承認済み（`approved`）フェーズの成果物に変更が必要になった場合は、以下の手順で差分承認を行う。
承認済みファイルを黙って書き換えてはならない。

1. 変更内容と理由をユーザーに提示し、変更に着手する合意を得る
2. 該当フェーズの `*.status` を `draft` に戻し、`changelog.md` に記録する（操作: `reopened`、変更理由を含める）
3. 成果物を修正し、通常どおり選択式の承認確認を提示して再承認を得る
4. 後続フェーズが承認済みの場合は波及を確認し、影響があれば同様に `draft` に戻して修正する
   （例: 要件変更が設計・テスト設計に波及する場合は 02 / 03 も差分承認の対象にする）
5. 実装済みコードに影響する場合は、テスト計画 CSV の該当ケースの修正・追加もセットで行う

### プロジェクト規約への昇格（ルール提案）

作業中に「今回の機能に限らずプロジェクト全体のルールにすべき」と気づいた事項
（例: 通信中 UI の標準方式、命名規則、共通部品の使い方、規約ファイルの **TBD** 項目の確定）は、
エージェントの判断で規約ファイルに書き込んだり、その場限りの独自判断で実装したりしない。

1. 提案内容・理由・反映先（`.claude/rules/` / `.claude/skills/` / `docs/specs/_templates/`）を提示し、
   `AskUserQuestion` で「**規約に採用 / 今回の機能限り / 不採用**」を確認する
2. **規約に採用**: 承認後に該当ファイルへ反映し、機能の変更と混ぜず**単独でコミット**する
   （例: `[SDD][rules] frontend 規約確定: 二重送信防止はボタン非活性方式`）
3. **今回の機能限り**: 規約ファイルには書かず、当該機能の `02-design.md` 等の成果物にのみ記載する
4. 規約への反映はフェーズ承認とは**別の承認**である（フェーズ承認に混ぜて暗黙に規約を変えない）

## 工数レポート（effort-report.md）

人手作業との比較で工数削減効果を測定するため、`docs/specs/<slug>/effort-report.md` を
テンプレート `docs/specs/_templates/effort-report.md` に沿って運用する。

- **見積（フェーズ 1 開始時）**: フェーズ 1 の成果物を作る際に「§1 人手想定工数」を記入する。
  「AI を使わず中堅エンジニア 1 名が手作業で同じ成果物を作った場合」をフェーズ別に見積もり、
  規模（節数・クラス数・テストケース数等）の根拠を添える。後続フェーズで規模の前提が
  大きく変わった場合は見積を更新し、変更理由を備考に残す
- **実績（各フェーズ承認時）**: 承認時（フェーズ 4 は完了時）に「§2 実績記録」へ
  着手・承認のチャット時刻と経過時間、差戻し回数を記入する。経過時間は人間の確認・
  承認待ちを含めた実時間とする
- **削減効果（フェーズ 4 完了時）**: 「§3 削減効果」に削減時間と削減率を算出して記入し、
  完了報告に含める
- 工数レポートは記録用であり、承認ゲートの対象外（status ファイルは作らない）

## フェーズ 1: 仕様整理

- 仕様を読み、不明点を `open-questions.md` に列挙して質問する
- 回答を反映し `01-requirements.md` を確定案にする
- Laravel 側とフロント（Blade / Vite / JS）の責務を明記する
- バリデーション（§7）・受け入れ条件（§11）・非機能要件（§9）に一意な ID
  （`VAL-xx` / `AC-xx` / `NFR-xx`）を付与する。ユースケース（§4 `UC-xx`）と合わせて、
  テスト計画フェーズのトレーサビリティ（要件カバレッジ表）で参照する

### 仮定の禁止（不明点はこのフェーズで詰め切る）

仕様の曖昧さ・不足を仮定で埋めて先へ進むことを禁止する。詰め切れていない仕様は
フェーズ 1 の中で解消してから承認確認を提示する。

- 仕様に書かれていないこと・複数の解釈があり得ることは、エージェントが**推測・仮定で
  埋めてはならない**（「【仮定】と明記すれば仮置きしてよい」という運用はしない）。
  必ず `open-questions.md` に質問として起票し、ユーザーの回答を得る
- 質問には選択肢や推奨案を添えてよい（回答しやすくするため）。ただし**回答を得る前に
  推奨案を `01-requirements.md` 本文に書いてはならない**
- `open-questions.md` の状態は `open`（未回答）/ `resolved`（回答済み・要件に反映済み）/
  `dropped`（ユーザーの判断で今回のスコープ外）で管理する
- **`open` が 1 件でも残っている間はフェーズ 1 の承認確認を提示しない**。全項目が
  `resolved` / `dropped` になるまで質問と反映を繰り返す
- ユーザーが「その前提で進めて」と明示的に指示した場合のみ、その内容を仮定ではなく
  **決定事項**として `open-questions.md` の回答欄に記録し（状態は `resolved`）、
  `01-requirements.md` 本文に反映する
- フェーズ 1 の承認確認には「open-questions: 全 ○ 件 resolved / dropped（open 0 件）」の
  報告を含める

## フェーズ 2: 設計

- `01-requirements.status` が `approved` の場合のみ着手
- Controller / Service / Model / Migration / Blade / Route / Job のクラス名・メソッド概要を記載
- 実装コードは書かない。人間がレビューできる粒度にする
- §6 影響範囲に、既存の画面・API・共通部品・テーブルへの波及を `IMPACT-xx` の ID 付きで洗い出す。
  影響なしと判断した項目も、検討した上での判断であることが分かる理由を書く（未検討の放置禁止）

### 独立レビュー（承認確認前に必須）

設計の見落としは下流（テスト設計・実装）でのやり直しコストが最も大きいため、
`02-design.md` を作成したら、**承認確認を提示する前に**独立レビューを行う。

1. **独立したレビュー用のサブエージェント**（`Agent` ツール・`subagent_type: sdd-design-reviewer`。
   定義: `.claude/agents/sdd-design-reviewer.md`。Read / Grep / Glob のみの読み取り専用）を起動する
2. レビュー対象として `01-requirements.md` / `02-design.md` のパスと、チェックリスト
   `docs/specs/<slug>/02-design-review-checklist.md` を渡し、全項目を検証させる。
   レビュアーは影響範囲（IMPACT）の記載を鵜呑みにせず、コードベースを実際に検索して裏取りする
3. 指摘のうち**機械的に修正可能なもの**（記載漏れ・ID 付与漏れ・理由の未記入等）は
   メインの AI がその場で修正する
4. **設計方針の変更を伴う指摘は自動修正せず**、承認確認の提示にレビュー結果のサマリを含めて
   人間に判断を仰ぐ
5. レビュー結果は `02-design-review-checklist.md` に記入し、成果物として残す

この独立レビューはフェーズ 3 の独立レビューと同様、人間の承認を代替しない。

## フェーズ 3: テスト設計

- `02-design.status` が `approved` の場合のみ着手
- 正常系・異常系・境界値・権限パターン・派生パターンを網羅する
- CSV はテスト種別ごとに分けて作成する:
  - Playwright E2E: `03-test-plan.csv`
  - PHPUnit: `03-test-plan-phpunit.csv`（Service 等の単体テストがある場合）
  - Vitest: `03-test-plan-vitest.csv`（JavaScript 単体テストがある場合）
  - E2E / PHPUnit / Vitest のケースを同じ CSV に混在させない
- `03-test-plan.csv` は 1 行 1 観点を原則とする。複数の入力項目、複数の
  エラー理由、複数のロール、複数の状態遷移など、対象・条件・期待結果が分かれる内容を
  1 ケースにまとめない。対象項目やエラー理由ごとに個別ケースへ分割する
- 業務ルール・入力検証・権限・状態判定は、異常系だけでなく正常に通る代表ケースも
  個別ケースとして記載する。原則として「OK ケース」と「NG ケース」を対で確認する
- 要件で許可されている操作・状態遷移（例: 承認・却下・返却）は全て列挙し、
  操作ごとに個別ケースを作成する。一部の操作だけ確認して残りを省略しない
- 条件分岐は取り得る値ごとに確認する。「制限あり・一致」「制限あり・不一致」だけでなく、
  「制限なし（NULL 等）」のような分岐値も正常系として個別ケースにする
- 編集（更新）機能がある場合は全項目について空入力の挙動を確認する:
  必須項目は「エラー + 元の値保持」、任意項目は「空値で上書き」（テンプレート
  `03-test-plan.md` の「更新系の標準観点」に従う）
- Playwright（E2E）を主、PHPUnit は Service 等の単体テスト、Vitest は JS 単体テストに限定
- CSV 列形式は `docs/specs/_templates/03-test-plan.csv` のヘッダー行に従う
- トレーサビリティ: `01-requirements.md` の UC / VAL / AC / NFR / 画面項目（§5.2）/ 権限（§8）の
  全項目に対応する Test ID が存在することを「要件カバレッジ表」（`03-test-plan.md`）で確認する。
  CSV の「出典」列には対応する要件側 ID を記載し、対応漏れがないことを承認依頼前に確認する
- 非機能要件（NFR-xx）が E2E で検証困難な場合は、PHPUnit のケースにするか、要件カバレッジ表の
  備考に「対応不可・理由・代替手段」を明記する（放置しない）
- `02-design.md` §6 で「影響あり」とした項目（`IMPACT-xx`）に対応する回帰確認のテストケースを
  `03-test-plan.md` の「回帰確認」で作成する
- 期待結果は具体的な値・文言・件数・画面状態で記載し、「正しく」「適切に」「問題なく」等、
  判定基準を含まない曖昧な表現を使わない（`03-test-plan.md` の「期待結果の書き方」参照）
- 各テストケースは他のケースの実行結果・実行順序に依存しないこと。前提条件はケース単独で
  再現可能な内容を記載し、共有データを使う場合はケース間で干渉しないようテストデータを分離する
  （`03-test-plan.md` の「独立性・再現性の標準観点」参照）
- Playwright は `fullyParallel: true` で実行されるため、単独実行の独立性だけでなく、
  **他のケースと同時に実行されても干渉しない**ことを確認する（`03-test-plan.md` の
  「並列実行安全性」、`testing-playwright` skill（`.claude/skills/testing-playwright/SKILL.md`）
  の「並列実行安全性」参照）

### 独立レビュー（承認確認前に必須）

`03-test-plan.md` と各 CSV を作成したら、**承認確認を提示する前に**、以下の手順で独立レビューを行う。

0. **機械的 lint を先に実行する（自動修正可能な問題を先に除去する）**:
   ```bash
   # テスト計画 CSV・カバレッジ表の静的検証
   npm run lint:sdd -- <slug>
   ```
   lint で ERROR が出た場合は機械的に修正してから次へ進む。WARN はサブエージェントレビューに
   持ち越して判断する（自動修正可能なものはその場で直す）。

1. 作成した本人（同一会話・同一コンテキストの AI）ではなく、**独立したレビュー用のサブエージェント**
   （会話コンテキストを引き継がない新規セッション、読み取り専用）を起動する
   （`Agent` ツール・`subagent_type: sdd-plan-reviewer`。定義: `.claude/agents/sdd-plan-reviewer.md`。
   ツールを Read / Grep / Glob に限定した読み取り専用構成のため、レビュー対象への意図しない
   書き込みを構成レベルで防げる。汎用の Explore エージェントは抜粋読みで全文検証に不向きなため
   使わない）
2. レビュー対象として `01-requirements.md` / `02-design.md` / `03-test-plan.md` / 各 CSV のパスを渡し、
   `docs/specs/<slug>/03-test-plan-review-checklist.md` の全項目を検証させ、項目ごとに
   ✓ / ✕ と理由を判定させる
3. サブエージェントの指摘のうち、**機械的に自動修正可能なもの**
   （カバレッジ表の空欄、曖昧な期待結果の言い換え、出典 ID の記載漏れ、CSV 粒度の分割漏れ等）は
   メインの AI がその場で修正する
4. 修正後、指摘が解消したかを再確認する（同じ観点を再度確認する。必要であればサブエージェントに
   再レビューさせる）
5. **業務ルールの解釈や仕様追加の要否など、判断が必要な指摘は自動修正せず**、承認確認の提示に
   レビュー結果のサマリ（残った指摘事項とその理由）を含めて人間に判断を仰ぐ
6. レビュー結果は `03-test-plan-review-checklist.md` に記入し、成果物として残す

この独立レビューは人間の承認を代替しない。あくまで承認確認の質を上げるための事前チェックであり、
「自己承認の禁止」（上記）の原則は変わらない。

## フェーズ 4: 実装・テスト

- `03-test-plan.status` が `approved` の場合のみ着手
- 実装 → PHPUnit（該当時）→ Vitest（該当時）→ Playwright の順で実行
- 自動修正は最大 3 イテレーション。解決しない場合は人間にエスカレーションする

### 実装完了の基準

以下をすべて満たした時点で実装完了とする。

| テスト | 基準 |
|--------|------|
| PHPUnit | 全件成功 + Service・Enum 等の担当範囲は原則 100%、全体 80% 以上（`--coverage --min=80`） |
| Vitest | 全件成功 + テスト対象 JS モジュールは 90% 以上（`npm run test:coverage`、閾値で強制） |
| Playwright | `03-test-plan.csv` の全ケースを実装し全件成功（コードカバレッジは対象外） |

- カバレッジ閾値に到達できない行がある場合は、テストを歪めて通すのではなく、
  理由を `03-test-plan.md` または PR に明記して許容する
