# ファイル編集時の文字化け事故防止（Windows）

対象: 常時適用。日本語（マルチバイト文字）を含むファイルを大量に扱うプロジェクトでの注意事項。

Windows 環境ではシェル経由のテキストパイプライン（PowerShell の `Get-Content | Set-Content`、
`echo` によるリダイレクト、`sed` / `awk` 等）を介した編集で、**サイレントに文字化け
（文字欠落・`?` への置換）が起きる**ことがある。一度発生すると気付きにくく、
原因調査・復旧に大きな時間を要する。

## 禁止事項

日本語（またはその他マルチバイト文字）を含むファイルに対して、Bash ツール経由での
以下のようなテキストパイプライン編集・作成は行わない:

- PowerShell の `Get-Content ... | Set-Content ...`（`-replace` を挟む場合も含む）
- `Get-Content ... | Out-File ...`
- `echo` / `"..."` のリダイレクト（`>` / `>>`）によるファイル作成・追記
- `sed` / `awk` 等によるテキスト置換パイプ

## 代わりに使うもの

- ファイルの新規作成・全文書き換え → `Write` ツール
- 部分編集（文字列置換） → `Edit` ツール
- 内容確認 → `Read` ツール
- どうしても Bash ツール経由が必要な場合（例: Git 履歴からの復元）は、テキストエンコーディング
  変換を経由しない方法を使う（例: `git show <ref>:<path> > out.txt`）。実行後は必ず `Read` で
  文字化けがないか確認する

## 大きなファイルの書き込みで文字化けを検知した場合

`Write` ツールでも、まれに大きなファイル・特定パスへの上書きで文字化けが再現することがある。
その場合は以下のワークアラウンドを使う:

1. 同内容を **一時ファイル**（例: `<元のパス>.new`）に `Write` で書き込む
2. `Read` で一時ファイルを読み、日本語が正しく書き込まれているか確認する
3. 問題なければ Bash ツールで `mv -f <元のパス>.new <元のパス>` を実行して上書き移動する
   （バイナリ移動のためエンコーディング変換が発生しない）

## 編集後の確認

日本語を含むファイルを編集した後は、`Read` ツールで該当箇所を読み直し、
文字化け（`�`、意図しない `?`、欠落）がないか確認する。特に一括置換・大量編集の直後は必須。
